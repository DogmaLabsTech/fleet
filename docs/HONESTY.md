# The honesty doctrine

A progress ring is a promise. When a dashboard says a project is **80% done**, a
stranger reads that as fact. Most dashboards earn that number by *asking the
project to grade itself* — a status field someone typed, a checkbox someone
ticked. The ring fills up because somebody **said so**, and nothing ever checks.

Fleet refuses to let a ring silently lie.

## Why "done" isn't enough

A milestone marked `done` is a **claim**. Claims have receipts: a commit, a PR, a
coverage file, a passing-test report. Fleet's job is to ask, for every claim, one
question — *is there a receipt, and does the receipt actually back the claim?* —
and then to **show the answer honestly in the ring**, never to hide a bare
assertion behind a confident green arc.

So Fleet sorts every milestone into one of four tiers and colours the ring by what
it found, not by what was claimed.

## The four tiers

| Tier | What it means | How a milestone earns it |
|---|---|---|
| **verified** | An independent repo artifact backs the claim. | `status: done` **and** a `verify` block whose read-only check **passes** (the file/glob/json the sidecar points at exists and matches). |
| **attested** | A cited receipt, taken on trust. | A `provenance` string (commit, PR, branch, URL) but **no** `verify` block — or a `verify` block Fleet couldn't read. Someone named a receipt; Fleet didn't open it. |
| **uncited** | No receipt at all. | `status: done` (or `in-progress`) with neither a passing `verify` nor any `provenance`. A bare assertion. |
| **contradicted** | Claimed done, but the receipt is missing or wrong. | `status: done` **and** a `verify` block whose check **fails** — the artifact is absent or the predicate is false. The ring will not pretend this is finished. |

The ring is split into arcs by tier so the **shape of your honesty** is visible at
a glance: a fat verified arc means receipts; a fat attested arc means trust; any
uncited slice or contradicted flag is a question you haven't answered yet.

## Fleet verifies by *reading*, never by *executing*

This is the load-bearing rule, and it is absolute.

- Fleet's verify engine (`engine/verify.py`) **only reads files** inside the
  repo's own directory. It checks that a file exists, that a glob matches, or that
  a value inside a JSON report satisfies a predicate.
- It **never runs your tests, your build, your linters, or any command.** It does
  not shell out, launch a process, or open a socket. There is no `subprocess`, no
  `os.system`, no network call anywhere in the verify path.
- Every path the sidecar points at is **sandboxed to the repo root.** A path that
  resolves outside the repo (`../../etc/passwd`) is treated as *no signal* and
  never read.
- JSON reads are **size-capped** so a hostile or runaway report can't be used to
  exhaust memory.

The practical consequence: **a `verify` check is something your CI, your test
runner, or your own coding agent already produced** — Fleet only inspects the
*leftover artifact*. The proof is generated by you; Fleet just looks at it. This
keeps Fleet's trust boundary identical to the rest of the tool: **100% local,
read-only, zero egress, zero execution.** Pointing Fleet at a repo can never run
code on your machine.

## Why this matters

A green ring that means "verified" and a green ring that means "I said so" are
two very different promises. Fleet keeps them different. The verified-vs-attested
split is the whole point: it lets a stranger trust the dashboard *because it tells
on itself* when the receipts aren't there.
